3. Understanding Pub/Sub
Pub/Sub = Publish/Subscribe, a cloud messaging system.
Publisher: streamer.py sends Wikipedia events to the cloud
Topic: live-wikimedia-stream receives and holds messages
Subscriber: BigQuery (Week 3) will read from the topic

ADC = Application Default Credentials
Set up in Week 1 via gcloud auth application-default login
Pub/Sub publisher automatically uses ADC to authenticate, no manual login needed.

4. Upgrading streamer.py
Added pubsub_v1 import from google.cloud
Added PROJECT_ID and TOPIC_ID constants
Initialized PublisherClient and topic_path
Serialized each event to JSON bytes before publishing
Pub/Sub only accepts bytes, not strings or dictionaries.
Added async callback function to handle confirmations in background

Synchronous vs Asynchronous Publishing:
Synchronous: Script waits for GCP confirmation before sending next message. Slow.
Asynchronous: Script sends message and immediately moves to next event. Fast.
Callback handles confirmations in the background without blocking.

5. Testing and Validation

Fixed by creating topic directly via gcloud CLI.

Low message rate of 1.2/second on first version.

Rate jumped to 23-26 messages/second after fix.

Week 3
Date: June 15th - 21st
-----------------------------------

1. Project Direction
Decided to focus on Real-Time Content Quality Tracker.
Core question: Is Wikipedia growing or shrinking right now?
Tracks whether edits add or remove content, across all
editor types (bot, registered, anonymous), in real time.

2. Three Schemas Designed
Planned Schema: design blueprint and goals, no data.
Landing Schema: 13 raw fields extracted from Wikimedia JSON,
filtered to type == "edit" only (excludes log events like
blocks/deletions since they lack the length field).
Transformation Schema: 18 fields, 11 inherited + 7 new
derived columns (edit_size, edit_direction, is_large_edit,
contributor_type, is_vandalism_signal, hour_of_day,
content_change_category).

3. GCP BigQuery Setup
Enabled BigQuery API on fpt-internship-2026.
Created dataset: wikimedia_data
bq mk --dataset --location=US fpt-internship-2026:wikimedia_data

Created three tables using JSON schema files:
- planned_schema (reference table)
- recentchange_landing (13 fields)
- recentchange_transformed (18 fields)
bq mk --table fpt-internship-2026:wikimedia_data.TABLE_NAME schema.json

4. Connected Pub/Sub to BigQuery Landing Table
Created subscription: wikimedia-to-bigquery
gcloud pubsub subscriptions create wikimedia-to-bigquery
  --topic=live-wikimedia-stream
  --bigquery-table=fpt-internship-2026:wikimedia_data.recentchange_landing
  --use-table-schema

Granted roles/bigquery.dataEditor to Pub/Sub service account
so it has permission to write to BigQuery.

5. Built Transformation SQL Query
Wrote INSERT INTO query in week3-bigquery/transformation_query.sql
Reads from recentchange_landing, computes 7 derived columns,
inserts into recentchange_transformed.
Used WHERE id NOT IN (...) to prevent duplicate inserts on
repeated runs.

6. Validation
Confirmed landing table row count increases in real time
while streamer.py runs (evaluation metric passed).
Confirmed transformation table populates correctly with
accurate derived columns.
Found genuine insight: commonswiki and wikidatawiki showing
the largest net content growth, driven mostly by bot edits.

Final Result for Week 3:
Evaluation metric passed - SELECT COUNT(*) increases in
real time while pipeline runs.

Issues/Challenges:
DML (INSERT INTO) queries require a billing account linked
to the GCP project, even within free tier limits.
Fixed by linking existing billing account:
gcloud billing projects link fpt-internship-2026
  --billing-account=ACCOUNT_ID
Set up a $50 budget alert with email notifications at
50%, 90%, 100% thresholds as a safety net.

Pub/Sub to BigQuery subscription initially failed with
invalid_argument errors. Root cause: streamer.py was
publishing the full raw Wikimedia JSON (including nested
fields like meta.dt and length.old/new) which did not match
the flat landing table schema.
Fixed by adding build_landing_record() function in
streamer.py to filter to edit-type events only and flatten
nested fields before publishing.

contributor_type regex initially showed zero anonymous
editors. Investigation revealed Wikipedia now masks
anonymous editors using a temporary account format
(~YEAR-XXXXX-XX) instead of raw IP addresses.
Fixed by adding a second regex pattern to detect this
format alongside the legacy IPv4 pattern.

Week 4
Date: June 22nd - 28th
-----------------------------------

1. Scope Clarified by Mentor
Two models needed: bot detection, high-value contributor.
Each needs an offline version and a real-time version.

2. New Features Added
is_english, day_of_week, comment_length, user_edit_count
day_of_week ended up useless - all data was one day, no variance.

3. High-Value Label Defined
registered + addition + large edit. ~5% positive rate, healthy.

4. Bot Detection Model Built
Logistic regression, behavioral features only, no contributor_type.

5. Major Bug: Stale Evaluation Cache
ML.EVALUATE() with no table reuses an old cached split.
Same score (0.7941) across 6 feature changes - impossible.
Fix: always pass an explicit table to ML.EVALUATE().
Real score: AUC-ROC 0.8784. Target was passed the whole time.

6. High-Value Model Built
First try: AUC-ROC 1.0 - a red flag, not a win.
Label leakage - features were literally part of the label formula.
Removed leaking fields, retrained leak-free.
Final: AUC-ROC 0.789 using namespace, is_english, hour_of_day,
comment_length, user_edit_count, minor.

7. Bot Model Validated Against Real Flag
82.85% agreement.
Missed bots: small repetitive editors, look human.
Missed humans: very active power users, look like bots.
Same root cause both ways - activity level overlap.

8. Real-Time Prediction Tested
Ran streamer.py, transformed, predicted on last 60 min of data.
Both models worked correctly on brand new edits.
Found bug: user_edit_count wasn't refreshed for new rows.
Fix: refresh_user_edit_count.sql, run after every transformation.

Final Results:
Bot detection: AUC-ROC 0.8784 - passed.
High-value contributor: AUC-ROC 0.789 - leak-free, close, documented.

Biggest Lessons:
Always use an explicit table with ML.EVALUATE().
A perfect score means check for leakage, not celebrate.
New columns need a permanent spot in the pipeline, not just a backfill.

1. Scope Clarified by Mentor
Three parts this week:
(A) Java microservice that alerts on predictions above 0.85 score.
(B) Continue training the models as new data arrives.
(C) Dashboard visualizing data per day, model weights changing,
and prediction performance (accuracy, recall, precision).

2. Understanding MLOps
MLOps = getting a model into production and keeping it working:
deployment, automation, monitoring, retraining.
Core problem it solves: models decay silently as the world changes
(data drift, concept drift). They don't crash, they just get worse.
Week 5 is the shift from "a model that works once" to "a live system."

3. Continuous Training Setup
BigQuery ML can't update a model in place - it retrains from scratch.
So "continuous training" = scheduled retraining on the growing dataset.
Built 03_scheduled_retrain.sql as the full chain:
transform new data -> refresh user_edit_count -> rebuild ml_dataset
with split -> retrain both models -> log metrics.
Scheduled it as a BigQuery Scheduled Query every 6 hours.

4. Metrics Logging
Created three tracking tables (01_create_metrics_tables.sql):
metrics_data_volume, metrics_model_weights, metrics_performance.
Used CREATE TABLE IF NOT EXISTS so history is never wiped.
Weights logged in long format (one row per feature per retrain)
so it charts cleanly and handles both models' feature sets.

5. Java Alert Service
Installed JDK 21 + Maven. Built alert-service as a Maven project.
Object-oriented, five single-responsibility classes:
Prediction (data), BigQueryClient (data access), AlertService
(the 0.85 threshold logic), AlertNotifier (output), Main (wiring).
Reads from a new bot_predictions view (ML.PREDICT on the bot model).
Threshold is a constructor parameter, not hardcoded - it's a
precision/recall dial, same tradeoff idea from Week 4 Slide 6.

6. Latency Fix
First run measured 542ms - over the 200ms target.
Root cause was NOT the logic - it was printing 5,679 alert lines
to the console (slow I/O). The spec means per-row evaluation.
Measured per-row instead: 0.00019ms per row across 26,794 rows.
Lesson: measure the thing the spec actually asks for, not a batch
of console prints.

7. Monitoring Dashboard
Self-contained HTML with Chart.js, three charts, bot/minor toggle:
records per day, performance over time, weights over time.
The final snapshot visibly jumped when fresh data arrived after
a power outage - continuous training made observable.

Final Result for Week 5:
Continuous retraining scheduled and logging snapshots every 6 hours.
Java service fires alerts on >0.85 predictions, ~0.0002ms per row
(far under 200ms target - passed).
Dashboard shows all three required metrics for both models.

Issues/Challenges:
Pipeline gap found: streamer kept the landing table fresh, but the
transformation step wasn't auto-running, so the transformed table
was 3 weeks stale. Continuous training would have retrained on old
data. Fixed by folding the full chain (transform + refresh + split)
into the scheduled job, not just the retrain.

refresh_user_edit_count only updated null rows. That's wrong for
continuous running - counts grow as users make new edits. Dropped
the IS NULL filter so it recalculates every row each cycle.

Java files saved to the wrong folder after a power outage (VS Code
closed before saving to src/main/java/...). Maven only compiles
files under src/main/java, so they were invisible. Moved them into
the correct package folder.

Latency looked like a fail (542ms) but was really console I/O, not
evaluation speed. Reframed to per-row latency, which is what the
spec actually asks for.

Known limitation: streamer runs locally, so a power outage paused
ingestion and several retrain snapshots are identical (models
correctly didn't move without new data). In production the streamer
would run on a server.
